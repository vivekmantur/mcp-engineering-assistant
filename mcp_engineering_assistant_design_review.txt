1. Project Overview
mcp-engineering-assistant is a GitHub-focused engineering assistant built around the Model Context Protocol (MCP). It enables developers to browse GitHub repository files, read file contents, run AI-assisted Python code review, generate non-destructive code-change proposals, edit those proposals interactively, and push approved changes back to GitHub Ś all from a single unified interface.

The system follows an approval-first philosophy: every MCP tool call must be explicitly approved by the user before execution. This is especially critical for write operations such as file pushes, providing a safe human-in-the-loop layer over all AI-driven actions.

1.1 Core Capabilities
Browse any GitHub repository&apos;s file tree
Read and display file content in a Monaco code editor
Request AI-powered Python code reviews backed by local policy documents
Generate complete revised code files based on review findings
Compare original vs. proposed code side-by-side in a diff modal
Edit AI-proposed code before pushing
Push committed changes back to GitHub with dual confirmation
Scan repositories and recent commits for basic security risks

2. System Architecture
The system is composed of three main layers that work together through a structured approval workflow:

Component
Details
MCP Server
Exposes engineering tools using FastMCP. Handles all GitHub API calls and AI-driven code review/generation via the Groq LLM.
FastAPI Bridge
HTTP bridge between the React frontend and the MCP server. Manages approval-based chat sessions, tool lifecycle, and state transformation.
React Frontend
Repository browser, Monaco code editor, chat approval panel, review output, code-change comparison modal, and push confirmation flow.

6. Key Module Reference
6.1 main.py
Entry point that starts the MCP server
Imports all tool and resource modules so their FastMCP decorators register with the shared singleton
Registered tools: GitHub browsing, file read, file push, code review, code change proposal, and security scan tools

6.2 bridge_api.py
FastAPI HTTP bridge between the React frontend and the MCP agent
Applies permissive CORS (allow_origins=["*"]) for local development
Exposes four endpoints used by the frontend:
POST /list-files Ś list repository files
POST /read-file Ś read a specific file
POST /agent-chat Ś send a message or deterministic shortcut
POST /agent-tool-decision Ś approve or reject a pending tool call

6.3 app/server/mcp_instance.py
Defines the shared FastMCP("EngineeringAssistant") singleton instance
All tools and resources import and attach to this shared instance via decorators

6.4 app/agents/mcp_chat_agent.py
Core approval-based agent orchestrator Ś the central coordination layer
Maintains in-memory CHAT_SESSIONS keyed by session ID
Discovers available MCP tools by spawning python -m main via MCP stdio transport
Sends messages to Groq with MCP tools exposed as function-calling tools
Supports deterministic shortcut messages to bypass LLM tool-choice drift:
__MCP_REVIEW_PYTHON_CODE__ Ś triggers review without LLM tool-selection
__MCP_PROPOSE_PYTHON_CODE_CHANGES__ Ś triggers code change proposal
__MCP_PUSH_GITHUB_FILE__ Ś triggers the push flow
Converts raw MCP tool results into UI-friendly state fields: files, selected_file, code, review, changed_code, pushed_file
Handles multiple messy MCP output formats: wrapped JSON, concatenated JSON objects, text blocks, structured content
Enforces one pending tool at a time for clear user approval UX

6.5 app/utils/github_client.py
Shared async GitHub helper layer used by all GitHub tools
get_github_file(owner, repo, path): calls Contents API, decodes base64 content, returns raw text
update_github_file(owner, repo, path, content, message, branch): reads current file SHA, base64-encodes updated content, uses PUT to commit

6.6 app/resources/*.md
Local policy/context files loaded by the review tool to guide AI judgment:
coding_standards.md Ś code style rules and structure expectations
security_guidelines.md Ś security risk patterns to flag
python_best_practices.md Ś Pythonic idiom and quality standards
review_constraints.md Ś hard constraints (preserve API behavior, routes, config; avoid unnecessary rewrites)


7. MCP Tool Catalog
All tools are registered with the shared FastMCP instance and are discoverable by the agent at runtime.

Tool Name
Category
Description
list_repository_files
GitHub
Lists root-level files and folders for a given owner/repo. Returns name, path, and type for each entry.
read_github_file
GitHub
Reads a specific file from GitHub. Returns name, path, and decoded content.
push_github_file
GitHub
Pushes edited content back to GitHub using the Contents API PUT. Accepts an optional branch parameter. Returns commit metadata.
list_my_repositories
GitHub
Lists all repositories accessible to the configured GITHUB_TOKEN.
review_python_code
Review
Loads local policy markdown resources and calls Groq to produce a structured code review. Review prompt explicitly forbids code examples in output.
propose_python_code_changes
Review
Generates a complete revised Python file based on the code and prior review. Strips <think> blocks and markdown fences from output.
scan_repository_security
Security
Scans root-level Python files for basic risk patterns: eval(, os.system(, hardcoded API-key-like strings.
scan_recent_commit_changes
Security
Fetches the latest commit from GitHub and scans changed Python files for the same risk patterns. Returns commit SHA, scanned files, and findings.

9. Approval-Based Execution Workflow
The approval-first design ensures no MCP tool executes without explicit human consent. The diagram below illustrates the full cycle:

(Insert screenshot from local machine here)

Key design properties of the approval workflow:
Only one pending tool exists per session at any time Ś the UI always shows a single, unambiguous decision
Deterministic shortcuts bypass LLM tool-selection for write operations, preventing accidental or drifted tool choices
Tool results are normalized into UI-friendly fields before updating React state
The push flow has two distinct confirmation layers: browser confirm() and the MCP approval card

10. Security and Safety Notes
Component
Details
GitHub Token Scope
GITHUB_TOKEN is stored server-side in .env only; never exposed to the frontend or browser.
Push Confirmation
Push operations require two confirmations: a browser confirm() dialog and an MCP approval card click.
Session Storage
Chat sessions are stored in-memory in the FastAPI process. Sessions reset on backend restart Ś no persistent database.
CORS Policy
Currently configured with allow_origins=["*"] for local development. Should be restricted in production.
User Authentication
No user authentication layer is present in the current implementation.
Code Edit Before Push
Users can review and modify AI-proposed code in the diff modal before committing any changes.
Security Scan Scope
The repository and commit security scanners perform basic pattern matching only (eval, os.system, API key patterns). They are not a substitute for a full SAST tool.
